# Project Constitution: hief

> Governing principles for the HIEF project — rules no agent or contributor may violate.
> Inspired by [GitHub Spec-Kit's /speckit.constitution](https://github.com/github/spec-kit).

## Identity

- **Name:** hief
- **Purpose:** A sidecar CLI for AI coding agents providing local code indexing, intent tracking, and evaluation tooling.
- **Architecture:** Rust binary + SQLite database + MCP server; no cloud dependencies in the critical path.

## Inviolable Rules

### Code Quality
- [ ] All public functions must have doc comments (`///`)
- [ ] No `.unwrap()` in production code paths — use `?` or `.expect("context message")`
- [ ] All error types must implement `std::error::Error` via `thiserror`
- [ ] No `unsafe` blocks without documented safety invariants in a `// SAFETY:` comment
- [ ] Functions exceeding 50 lines must be refactored into smaller helpers

### Architecture
- [ ] All database access goes through `crate::db::Database` — no raw SQL elsewhere
- [ ] No direct `std::fs` calls outside of `crate::docs` and `crate::index`
- [ ] CLI parsing lives exclusively in `src/cli/`; business logic belongs in domain modules
- [ ] MCP tool handlers must remain thin — delegate to domain crates, never implement logic inline

### Dependencies
- [ ] Minimize dependency count; prefer `std` and established crates (e.g., `serde`, `tokio`, `clap`)
- [ ] Pin grammar/parser crate versions to avoid `cc` conflicts
- [ ] Async runtime: `tokio` only — never `async-std`
- [ ] Prefer `reqwest` behind the `embeddings` feature flag to keep the binary lean

### Process
- [ ] Every code change must trace to an approved intent (`hief graph create`)
- [ ] No agent may approve its own intent — a human reviewer must sign off
- [ ] Golden set regressions (`hief eval run --ci`) block merge
- [ ] All PR descriptions must indicate which intent ID is addressed

## Anti-Patterns

- **No silent failures** — errors must surface, not be swallowed with `let _ = ...`
- **No business logic in `main.rs`** — route to domain modules immediately after CLI parsing
- **No raw `println!` in domain code** — use `tracing::info!/warn!/error!`
- **No hardcoded file paths** — resolve everything relative to `project_root`
